# ZentroMeet Mobile — Native iOS Build on Codemagic (no EAS)

How the `ios-production` workflow builds, signs, and (optionally) ships the
ZentroMeet iOS app **natively on a Codemagic macOS machine** — no EAS Build,
no `EXPO_TOKEN`. The app stays a normal Expo / React Native (SDK 56) project;
Codemagic just runs the native toolchain (`expo prebuild` → CocoaPods → Xcode
archive → App Store export → TestFlight) the same way EAS used to.

> Companion to [`CODEMAGIC.md`](CODEMAGIC.md) (the full Android + iOS runbook).
> This file is the deep dive on the **native iOS** pipeline and exactly what
> the operator must configure in the Codemagic UI before it can run.

---

## 1. What changed and why

The old `ios-production` workflow ran on a cheap Linux box and **delegated the
real compile to EAS** (`eas build --platform ios`), then downloaded the `.ipa`.
That required `EXPO_TOKEN` and the whole EAS build/queue/credentials stack, and
it failed whenever `EXPO_TOKEN` was missing or EAS auth drifted.

The workflow now compiles iOS **entirely on Codemagic's Apple-silicon mac**:

```
npm ci
  → tsc / expo-doctor / unit tests          (quality gates, fail fast)
  → expo export --platform ios              (validate the JS bundle compiles)
  → expo prebuild --platform ios --clean    (managed → native ios/ project)
  → pod install                             (creates the .xcworkspace)
  → discover workspace + scheme             (verified, not hardcoded)
  → xcode-project use-profiles              (apply Apple signing — automatic)
  → xcode-project build-ipa                 (Xcode archive + App Store export)
  → validate the .ipa                       (hard gates, below)
  → app-store-connect publish --testflight  (HELD behind PUBLISH_TO_TESTFLIGHT)
```

**Expo is NOT removed and the project is NOT manually ejected.** `ios/` stays
git-ignored and is regenerated by `expo prebuild` on every build (managed
workflow preserved).

## 2. Machine + toolchain

| Setting | Value | Why |
|---|---|---|
| `instance_type` | `mac_mini_m2` | Native iOS compilation requires macOS + Xcode. Linux cannot build iOS. |
| `xcode` | `26.4` | Minimum for Expo SDK 56 / RN 0.85 (min iOS 16.4); also Codemagic's default image. Avoid the Xcode 27 beta — SDK 56 build-from-source fails on it ([expo/expo#46876](https://github.com/expo/expo/issues/46876)). |
| `node` | `22.13.0` | Matches the Android workflows; Expo SDK 56 / RN 0.85 requires Node ≥ 20.19.4. |
| `cocoapods` | `default` | Codemagic's bundled CocoaPods works with RN 0.85 / SDK 56 — no manual `gem install`. |
| `working_directory` | `mobile` | The app lives at `mobile/` in the monorepo; all scripts + artifact globs resolve from there. |

## 3. Code signing — automatic, via the App Store Connect API key integration

The workflow uses **Codemagic automatic signing**. Two pieces:

```yaml
integrations:
  app_store_connect: zentromeet_asc_api_key   # the Codemagic UI integration NAME

environment:
  ios_signing:
    distribution_type: app_store               # App Store / TestFlight distribution
    bundle_identifier: com.zentromeet.app
```

With the integration present + `ios_signing` set, Codemagic **fetches or
creates the Apple Distribution certificate and the App Store provisioning
profile** for `com.zentromeet.app` from Apple at build time, then
`xcode-project use-profiles` applies them to the generated Xcode project.

Nothing sensitive is in git — **no `.p8`, `.p12`, certificate, provisioning
profile, password, Issuer ID, or Apple ID** is committed. `ios/` (which would
contain the applied profile) is git-ignored.

### Entitlements (only what the app genuinely uses)

The App Store provisioning profile (created/fetched by Codemagic) must include
exactly these two entitlements, because the app uses them:

| Entitlement | Why | Evidence |
|---|---|---|
| `aps-environment` | **Remote** push notifications | `getExpoPushTokenAsync()` + token registration to `POST /api/mobile/push-tokens` (`src/hooks/usePushNotifications.ts`). Needs an APNs Auth Key (.p8) configured on the Apple side. |
| `com.apple.developer.associated-domains` | Universal Links | `ios.associatedDomains = ["applinks:app.zentromeet.com"]` (`app.json`). Requires `apple-app-site-association` served at `https://app.zentromeet.com/.well-known/apple-app-site-association`. |

**Deliberately NOT enabled** (audited, not used): Sign in with Apple
(`com.apple.developer.applesignin`), background modes / `UIBackgroundModes`,
and Keychain Sharing (`keychain-access-groups`) — `expo-secure-store` is used
without a shared access group, so it does not need the Keychain Sharing
entitlement.

These entitlements are derived by `expo prebuild` from `app.json` (the
`expo-notifications` plugin + `ios.associatedDomains`). When Codemagic
auto-manages signing, ensure the App Store profile for `com.zentromeet.app`
includes Push Notifications + Associated Domains capabilities (Codemagic
creates them automatically when using the API-key integration; if you manage
profiles manually instead, enable those two capabilities on the App ID).

## 4. TestFlight upload — held behind a flag

Codemagic's declarative `publishing.app_store_connect.submit_to_testflight` is
a **boolean** and cannot be driven by an environment variable. To honor "build
the signed IPA first, hold the upload behind a flag, do not auto-submit during
the config rollout," the upload is a **script step** using Codemagic's own CLI:

```yaml
vars:
  PUBLISH_TO_TESTFLIGHT: "false"   # default = build + validate IPA, DO NOT upload
```

```bash
# Step "Upload to TestFlight (flag-gated)":
if [ "$PUBLISH_TO_TESTFLIGHT" != "true" ]; then
  echo "skipping TestFlight upload"; exit 0
fi
app-store-connect publish \
  --path "build/ios/ipa/*.ipa" \
  --testflight \
  --max-build-processing-wait 20
```

- `--testflight` uploads the build and submits it to **TestFlight beta review
  only**. We deliberately **omit `--app-store`**, so it **never** submits for
  App Store review.
- This is **Codemagic's App Store Connect publishing** (the `app-store-connect`
  CLI from codemagic-cli-tools) — **NOT** `eas submit`.
- Auth reads `APP_STORE_CONNECT_ISSUER_ID` / `APP_STORE_CONNECT_KEY_IDENTIFIER`
  / `APP_STORE_CONNECT_PRIVATE_KEY` from the `app_store_credentials` group.

**Default behavior (`PUBLISH_TO_TESTFLIGHT="false"`):** the workflow produces
and validates a signed App Store IPA as a downloadable Codemagic artifact and
**uploads nothing**. Flip the flag to `"true"` in the Codemagic UI only when
you actually intend to ship to TestFlight.

> Alternative (fully-declarative) once you want every `main` build to publish:
> replace the gated script with a `publishing.app_store_connect` block
> (`auth: integration`, `submit_to_testflight: true`, `submit_to_app_store:
> false`). It removes the flag but uploads on every successful build.

## 5. IPA validation (hard gates)

The "Validate IPA" step fails the build unless **all** hold:

| Check | How |
|---|---|
| An IPA exists | `ls build/ios/ipa/*.ipa` |
| IPA is a believable size | `≥ MIN_IPA_BYTES` (3 MB; a real RN IPA is tens of MB) — catches an empty/tiny zip |
| Correct bundle id | `CFBundleIdentifier == com.zentromeet.app` |
| Build number present | `CFBundleVersion` non-empty |
| Universal (iPhone + iPad) | `UIDeviceFamily` contains both `1` and `2` |
| Distribution signing | `codesign -dvv` Authority is **Apple Distribution** |
| App Store export (not ad-hoc/dev) | embedded profile has **no** `ProvisionedDevices` and `get-task-allow=false` |

---

## 6. Workspace + scheme discovery (not hardcoded)

The `.xcworkspace` only exists after `pod install`. The "Discover Xcode
workspace + scheme" step finds `ios/*.xcworkspace`, lists the workspace's
shared schemes via `xcodebuild -list -json`, picks the one matching the
generated `.xcodeproj` name (expected **`ZentroMeet`**, derived from
`expo.name`), and exports `XCODE_WORKSPACE` / `XCODE_SCHEME` to `$CM_ENV` for
the build step. If the name ever changes, the workflow adapts automatically.

## 8. Build number

The workflow uses the **current** `ios.buildNumber` from `app.json`
(currently `"11"`) as-is — it does **not** auto-increment. Bump it by hand in
`app.json` before a real TestFlight cut so each upload has a unique build
number (App Store Connect rejects a reused `CFBundleVersion` for the same
marketing version).

---

## 9. eas.json — retained as the optional manual fallback only

`mobile/eas.json` is **kept**. **No Codemagic workflow uses it** anymore
(Android builds via Gradle, iOS via the native pipeline above). Its only
remaining consumers are the optional local scripts in `mobile/package.json`
(`npm run build:ios:preview`, `build:android:production`, `submit:ios`, etc.),
which call `eas-cli` directly for a manual/laptop build. Deleting `eas.json`
would break those scripts, so it stays — purely as the optional EAS escape
hatch, not part of CI.

## 10. What the OPERATOR must configure in the Codemagic UI

The workflow **cannot build until these exist.** (`READY TO TRIGGER iOS BUILD`
stays **NO** until they are confirmed.)

1. **App Store Connect API key (integration)** — Codemagic → Teams →
   `<team>` → **Team integrations** → **Developer Portal** → Connect. Upload
   the App Store Connect API key (`.p8` private key) + **Issuer ID** + **Key
   ID** (App Store Connect → Users and Access → Integrations → App Store
   Connect API; the key needs **App Manager** access). **Name the integration
   `zentromeet_asc_api_key`** (must match `integrations.app_store_connect` in
   the YAML — or change the YAML to your chosen name). This single key drives
   both automatic signing and the TestFlight upload.

2. **Apple Distribution certificate** — created/fetched automatically by
   Codemagic from the API key at build time. No manual upload needed when using
   the integration. (Only if you switch to manual signing would you upload a
   `.p12`.)

3. **App Store provisioning profile for `com.zentromeet.app`** — also
   created/fetched automatically. Ensure the App ID has **Push Notifications**
   and **Associated Domains** capabilities enabled (see §3). The App Store
   Connect app record for `com.zentromeet.app` must already exist — **do not
   create a duplicate**.

3b. **APNs Auth Key registered with EXPO (required for push to actually deliver).**
   The backend sends pushes via the **Expo Push API** using ExponentPushTokens, so
   APNs delivery depends on an **APNs Auth Key (.p8)** registered with **Expo** for
   this project (`extra.eas.projectId` in app.json) — NOT just the Codemagic signing
   capability. Apple Developer → Certificates, Identifiers & Profiles → **Keys** →
   enable **Apple Push Notifications service** → create the `.p8`; then register it
   with Expo for the iOS credentials (`eas credentials` → iOS → Push Notifications:
   APNs Key, or the Expo dashboard). Codemagic's automatic signing only provisions
   the Push Notifications *capability/entitlement* — it does **not** configure APNs
   delivery. The same APNs key is reused across builds; rotating the signing API key
   does not rotate it. **Never** commit or print the `.p8` in git or CI logs.

4. **Reference names used in the YAML** —
   - Integration: `zentromeet_asc_api_key`
   - Signing bundle id: `com.zentromeet.app`
   - Env groups: `zentromeet_api`, `app_store_credentials`

5. **Environment variable groups** —
   - **`zentromeet_api`** → `EXPO_PUBLIC_API_BASE_URL = https://app.zentromeet.com`
     (inlined into the JS bundle at build time).
   - **`app_store_credentials`** (Secure) → `APP_STORE_CONNECT_ISSUER_ID`,
     `APP_STORE_CONNECT_KEY_IDENTIFIER`, `APP_STORE_CONNECT_PRIVATE_KEY`
     (the same API key's values; read by `app-store-connect publish`). Only
     needed when you flip `PUBLISH_TO_TESTFLIGHT` to `true`.

6. **Apple Team ID** — supplied automatically by the API-key integration for
   automatic signing; you do not set it in YAML. (Have it handy for the App
   Store Connect record.)

7. **Publishing toggle** — `PUBLISH_TO_TESTFLIGHT` defaults to `"false"`
   (build + validate the IPA, no upload). Set it to `"true"` in the Codemagic
   UI for the `ios-production` workflow **only after** steps 1 + 5 are done and
   verified. It uploads to TestFlight only — never App Store review.

8. **Configuration file path** — Project Settings → Build → `codemagic.yaml`
   (repo root; monorepo). (Already required for the Android workflows.)

> `EXPO_TOKEN` / the `expo_credentials` group are **NOT required** by this
> workflow. (`expo_credentials` is still referenced by `android-production` in
> a non-fatal legacy step; that is unrelated to iOS and can be cleaned up
> separately.)

## 11. First native iOS build — checklist

1. Confirm the App Store Connect API key integration `zentromeet_asc_api_key`
   exists (§10.1) and the `zentromeet_api` group is set.
2. Keep `PUBLISH_TO_TESTFLIGHT="false"` for the first run.
3. Manually start `ios-production` in the Codemagic UI (any branch).
4. Wait for the signed IPA artifact; download it. Verify the "Validate IPA"
   step passed (universal, App Store-signed, correct bundle id + build number).
5. Configure the `app_store_credentials` group (§10.5).
6. Set `PUBLISH_TO_TESTFLIGHT="true"` and re-run to upload to TestFlight.
7. App Store Connect → TestFlight shows the build in ~10–30 min after Apple
   finishes processing.
