# ZentroMeet Mobile — Android 16 KB Page-Size Compliance

Google Play (from **Nov 1, 2025**) rejects updates that target Android 15+
(API 35+) unless every native `.so` library is aligned for **16 KB memory
pages**. ZentroMeet hit this with:

> "Your app does not support 16 KB memory page sizes."

This document records the fix shipped for **versionCode 20 / versionName 0.3.1**.

---

## Root cause

The non-compliant libraries were the **React Native / Hermes core**
(`libhermes.so`, `libreactnative.so`, `libjsi.so`, …). On **Expo SDK 52 /
React Native 0.76.9** those ship as prebuilt 4 KB-aligned binaries, and
[Expo's official guidance](https://github.com/expo/fyi/blob/main/android-16kb-page-sizes.md)
is unambiguous: there is **no supported way** to make SDK 52 produce
16 KB-aligned core libraries —

> "React Native supports 16 KB page sizes since version 0.77. You need to
> upgrade to an Expo SDK that includes React Native 0.77, which means Expo
> SDK 53."

## Fix: upgrade Expo SDK 52 → 56

| | Before | After |
|---|---|---|
| Expo SDK | `~52.0.11` | `^56.0.12` |
| React Native | `0.76.9` | `0.85.3` |
| React | `18.3.1` | `19.2.3` |
| Architecture | Old (`newArchEnabled: false`) | **New Architecture** (mandatory in RN 0.85) |
| Android NDK | (SDK 52 default) | **27.1.12297006** (16 KB-aligned ELF) |
| Android Gradle Plugin | (SDK 52 default) | **8.12.0** (16 KB zip-align) |
| `targetSdkVersion` | 35 | 35 (unchanged — runtime behavior preserved) |
| `compileSdkVersion` | 35 | 36 (SDK 56 dependency requirement) |

SDK 56 + **NDK 27 + AGP 8.12** produces 16 KB-aligned native libraries by
default, so the AAB is compliant with no manual linker flags.

### Notable dependency bumps (via `expo install --fix`)
`react-native-reanimated` 3.16 → **4.3.1** (now requires
`react-native-worklets`, added), `react-native-screens` 4.4 → 4.25,
`react-native-safe-area-context` 4.12 → 5.7,
`@react-native-async-storage/async-storage` 1.23 → 2.2,
`react-native-svg` 15.8 → 15.15, `expo-router` 4.0 → 56.2, `@types/react`
18 → 19, `typescript` 5.3 → 6.0.

## How to build & upload the production AAB

Both paths read `versionCode`/`versionName` from `app.json`
(`appVersionSource: local`, no auto-increment) — already bumped to **20 /
0.3.1**.

### Primary — Codemagic `android-production`
Pushing to **`main`** triggers it (`expo prebuild` → `./gradlew
bundleRelease`). It now:
- runs on **Node 22.13.0** (RN 0.85 requires Node ≥ 20.19.4),
- gates on **16 KB alignment** (see below) before publishing the artifact.

Artifact: **`android/app/build/outputs/bundle/release/app-release.aab`**
(the `android/` dir is generated by prebuild), downloadable from the
Codemagic build's Artifacts section.

### Fallback — EAS (manual/laptop)
```bash
cd mobile
npm run build:android:production   # eas build --profile production --platform android
```

> **Do not** submit to Play automatically. Upload the AAB to the Play
> Console manually (or run `npm run submit:android`) only after review.

## 16 KB validation (proof for upload)

`scripts/check-16kb-alignment.sh` unpacks an `.aab`/`.apk`, reads every
native `.so`'s ELF `LOAD`-segment alignment, and **fails if any is < 16 KB**.

- **In CI:** the Codemagic `android-production` workflow runs it as a hard
  gate after the build — a non-compliant bundle can never be published. The
  per-library MIN-ALIGN table is printed to the build log.
- **Locally**, on a downloaded AAB:
  ```bash
  bash scripts/check-16kb-alignment.sh ~/Downloads/app-release.aab
  ```
  Expected output ends with:
  `RESULT: ✓ All N native libraries are 16 KB aligned. Play-compliant.`

(Equivalent official tool: Google's `check_elf_alignment.sh` from the NDK
samples — same ELF-alignment check.)

## Caveats

- **iOS is intentionally untouched** (Android-only scope). The
  `ios-production` Codemagic workflow still pins `node: 20.17.0` and
  `xcode: "16.2"`, both of which are **too old for RN 0.85** and must be
  raised before the next iOS build. No iOS work was done here.
- After a fresh checkout, run `npm install` in `mobile/` (CI uses `npm ci`).
  An `.npmrc` pins `legacy-peer-deps=true` so installs are consistent.
- Rollback safety nets `package.json.sdk52.bak`, `package-lock.json.sdk52.bak`,
  and `app.json.sdk52.bak` are left in `mobile/`; delete them once the SDK 56
  build is confirmed on a device.
